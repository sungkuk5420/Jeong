### 인증 정책 비교

| 앱 | 비로그인 탐색 | 로그인 필요 기능 | 소셜 로그인 | 온보딩 방식 |
|---|:---:|---|---|---|
| **TripAdvisor** | ✅ 전체 | 리뷰 작성, 저장, 포럼 | Google, Facebook, Apple | Browse First |
| **Yelp** | ✅ 전체 | 리뷰, 사진 업로드, 북마크 | Google, Facebook, Apple | Browse First |
| **Google Maps** | ✅ 전체 | 리뷰, 리스트 저장, 기여 | Google 계정 | 자동 연동 |
| **Klook** | ✅ 탐색만 | 예약, 결제, 위시리스트 | Google, Facebook, Apple | Browse First |
| **GetYourGuide** | ✅ 탐색만 | 예약, 결제 | Google, Facebook, Apple | Browse First |
| **MangoPlate** | ✅ 전체 | 리뷰, 맛집 등록 | Kakao, Naver, Google | Browse First |
| **VisitKorea** | ✅ 전체 | 일부 맞춤 기능 | 제한적 | 완전 개방 |

### 공통 패턴: Browse First, Register Later

**모든 성공적인 여행/맛집 앱은 비로그인 탐색을 허용합니다.**

- 핵심 콘텐츠(검색, 조회, 지도)는 100% 개방
- 사용자 참여 기능(리뷰, 저장, 예약)에서만 로그인 요구
- 가입을 강제하지 않고, 가치를 먼저 보여준 후 자연스럽게 유도

## 2. 회원가입/로그인 필수 vs 선택 - 장단점

### 방식 A: 필수 로그인 (Mandatory Auth)

| 장점 | 단점 |
|------|------|
| 사용자 데이터 즉시 수집 가능 | **회원가입 단계에서 60~80% 이탈** |
| 개인화 서비스 바로 제공 | 첫 인상에서 "귀찮은 앱" 인식 |
| 스팸/어뷰징 차단 용이 | 앱스토어 리뷰에서 부정적 평가 |
| 푸시 알림 타겟팅 가능 | 외국인은 인증 수단 제한으로 더 높은 이탈 |

### 방식 B: 완전 개방 (No Auth Required)

| 장점 | 단점 |
|------|------|
| 진입 장벽 제로 | 사용자 식별 불가 |
| 높은 초기 리텐션 | 개인화 서비스 제공 불가 |
| 빠른 사용자 확보 | 스팸/저품질 콘텐츠 관리 어려움 |
| 외국인 접근성 최고 | 재방문 유도 수단 부족 (푸시 등) |

### 방식 C: 하이브리드 (권장) - Lazy Registration

| 장점 | 단점 |
|------|------|
| **가치를 먼저 경험 → 자연스러운 전환** | 구현 복잡도 증가 |
| 높은 초기 리텐션 + 점진적 데이터 수집 | 게스트 → 회원 전환 설계 필요 |
| 외국인 친화적 | 비로그인 데이터 → 로그인 계정 연결 로직 |
| 업계 표준 (Best Practice) | - |

## 3. 외국인 대상 특수 고려사항

### 소셜 로그인 우선순위

| 순위 | 인증 수단 | 이유 | 적용 |
|:---:|---|---|:---:|
| 1 | **Google** | 글로벌 점유율 1위, Android 기본 계정 | **필수** |
| 2 | **Apple** | iOS 사용자 필수 (App Store 정책), 선진국 점유율 높음 | **필수** |
| 3 | **Email/Password** | 소셜 로그인 거부 사용자 대안 | **필수** |
| 4 | Facebook/Meta | 동남아·유럽 사용자층 | 선택 |
| 5 | KakaoTalk | 한국 거주 외국인에게 유용 | 후순위 |
| - | ~~전화번호 인증~~ | ~~해외번호 수신 불가, 로밍 비용~~ | **제외** |

### 전화번호 인증을 피해야 하는 이유

- 외국인 관광객: 한국 번호 없음, 로밍 시 SMS 수신 불안정
- 일부 국가에서 SMS 인증 자체가 비용 발생
- **대안**: Email OTP 또는 Magic Link

### 언어/지역별 고려사항

| 지역 | 선호 인증 | 참고 |
|------|----------|------|
| 미국/유럽 | Google, Apple | GDPR 컴플라이언스 필수 |
| 동남아 | Google, Facebook | Facebook 사용률 높음 |
| 일본 | Apple, Google | LINE 연동 고려 가능 |
| 중국 | WeChat, Alipay | 별도 시장 진출 시 고려 |

### 개인정보보호 규제

| 규제 | 적용 대상 | 핵심 요구사항 |
|------|----------|-------------|
| **GDPR** (EU) | EU 시민 대상 서비스 | 명시적 동의, 데이터 삭제권, 최소 수집 |
| **PIPA** (한국) | 한국에서 운영하는 서비스 | 개인정보 수집·이용 동의, 제3자 제공 동의 |
| **CCPA** (미국 CA) | 캘리포니아 거주자 | 데이터 판매 거부권, 삭제권 |

**필수 조치:**
- 개인정보 처리방침 다국어(영/한) 제공
- 수집 항목 최소화 (이름, 이메일 정도)
- 회원 탈퇴 시 데이터 완전 삭제 기능
- 마케팅 동의는 별도 opt-in

## 5. Jeong 앱 추천 인증 전략

### 핵심 원칙: **Lazy Registration + Progressive Profiling**

> "가치를 먼저 보여주고, 필요할 때만 가입을 요청한다"

### 기능별 접근 권한 매트릭스

#### Tier 1: 완전 개방 (비로그인)
| 기능 | 설명 |
|------|------|
| 맛집/관광지 검색 | 키워드, 카테고리, 지역 검색 |
| 상세 정보 조회 | 위치, 영업시간, 메뉴, 사진 |
| 지도 보기 | 주변 맛집/관광지 지도 표시 |
| 리뷰 읽기 | 다른 사용자 리뷰 열람 |
| 카테고리 탐색 | 한식, 카페, 관광 등 카테고리 브라우징 |
| 기본 필터/정렬 | 거리순, 평점순, 인기순 |
| 앱 설정 | 언어 변경, 다크모드 |

#### Tier 2: Soft Wall (로그인 유도, 회원가입 페이지로 안내)
| 기능 | 유도 문구 예시 |
|------|-------------|
| 즐겨찾기/북마크 저장 | "Save this place? Sign in to keep your favorites" |
| 맞춤 추천 | "Get personalized picks — sign in for free" |
| 방문 기록 | "Track your visited places" |
| 여행 코스 만들기 | "Create your own Korea itinerary" |

#### Tier 3: Hard Wall (로그인 필수)
| 기능 | 이유 |
|------|------|
| 리뷰 작성 | 스팸 방지, 품질 관리 |
| 사진 업로드 | 어뷰징 방지 |
| 신규 장소 등록/제안 | 데이터 품질 보호 |
| 다른 사용자와 소통 | 신원 확인 |
| 프로필 관리 | 개인 정보 |

### 온보딩 플로우 설계

│              앱 최초 실행                          │

## 6. 구체적 정책 제안서

### 6.1 인증 정책 요약

| 항목 | 정책 |
|------|------|
| **기본 접근** | 게스트 모드로 핵심 기능 80% 사용 가능 |
| **로그인 방식** | Google, Apple, Email (3종) |
| **전화번호 인증** | 사용하지 않음 |
| **필수 수집 정보** | 이메일, 닉네임 (최소화) |
| **선택 수집 정보** | 프로필 사진, 국적, 관심사 |
| **자동 로그인** | 토큰 기반 (Refresh Token 30일) |
| **회원 탈퇴** | 앱 내에서 즉시 가능, 데이터 완전 삭제 |

### 6.2 Progressive Profiling 단계

| 단계 | 시점 | 수집 정보 |
|------|------|----------|
| 0단계 | 게스트 | 없음 (디바이스 ID만) |
| 1단계 | 최초 가입 | 이메일, 닉네임 |
| 2단계 | 첫 리뷰 작성 시 | 국적/언어 (선택) |
| 3단계 | 맞춤 추천 요청 시 | 음식 선호도, 관심 카테고리 |

### 6.3 소셜 로그인 구현 사양

| 항목 | Google Sign-In | Apple Sign-In | Email |
|------|:---:|:---:|:---:|
| **우선순위** | 1순위 | 2순위 | 3순위 |
| **Android** | ✅ Google One-Tap | ✅ (웹뷰) | ✅ |
| **iOS** | ✅ | ✅ (필수) | ✅ |
| **수집 데이터** | 이메일, 이름, 프로필사진 | 이메일, 이름 | 이메일 |
| **참고** | 가장 높은 전환율 | Apple 정책상 iOS 필수 | 대안 수단 |

> **Apple 정책 주의**: 앱에 소셜 로그인(Google 등)을 제공하면 Apple Sign In도 반드시 함께 제공해야 합니다.

### 6.4 기술 구현 가이드라인

```dart
// 인증 상태 3단계
enum AuthStatus {
  guest,          // 비로그인 (디바이스 UUID로 식별)
  authenticated,  // 로그인 완료 (기본 프로필)
  verified,       // 프로필 완성 (리뷰 작성 가능)
}

// 기능 접근 권한 체크
enum FeatureAccess {
  open,      // Tier 1: 누구나
  softWall,  // Tier 2: 로그인 유도 (바텀시트)
  hardWall,  // Tier 3: 로그인 필수 (페이지 이동)
}
```

### 6.5 KPI 목표

| 지표 | 목표 |
|------|------|
| 게스트 → 회원 전환율 | 15~25% (30일 내) |
| 회원가입 완료율 (가입 시작 기준) | 70% 이상 |
| Day-7 리텐션 (게스트 포함) | 15% 이상 |
| Day-30 리텐션 | 5% 이상 |
| 소셜 로그인 비율 | 80% 이상 (Google+Apple) |

### 6.6 로드맵

| 단계 | 내용 | 시기 |
|------|------|------|
| **Phase 1** | Google + Apple + Email 로그인, 게스트 모드, Tier 1~3 분리 | MVP |
| **Phase 2** | Progressive Profiling, 맞춤 추천, 푸시 알림 | v1.1 |
| **Phase 3** | A/B 테스트 (Soft Wall 문구/타이밍), Facebook 로그인 추가 | v1.2 |

## 7. 최종 결론

### Jeong 앱에 최적인 전략: **하이브리드 (Lazy Registration)**

```
핵심 원칙:
1. 가입 없이 맛집/관광지 정보를 자유롭게 탐색 가능
2. "저장하기", "리뷰 쓰기" 등 참여 기능에서 자연스럽게 가입 유도
3. Google/Apple 원탭 로그인으로 가입 허들 최소화
4. 전화번호 인증 절대 사용하지 않음 (외국인 장벽)
5. 수집 정보 최소화 → 점진적으로 프로필 완성
```

**이 전략을 선택하는 이유:**
- 외국인 사용자는 "설치 → 바로 정보 확인"을 기대함
- 강제 가입 시 60~80% 이탈은 초기 성장에 치명적
- TripAdvisor, Yelp, Klook 등 글로벌 성공 앱 모두 이 방식 채택
- 게스트 모드로 앱의 가치를 먼저 증명한 후 전환하는 것이 리텐션에 유리
